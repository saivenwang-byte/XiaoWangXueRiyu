## 一、本次全量对账结果（仓库内）

在**当前工作区**执行（不要手查喇叭）：

```bat
批量检查语音包.bat
python scripts/report-tts-audit.py --write-registry
```

| 指标 | 要求 | 说明 |
|------|------|------|
| 需朗读条数 | 与 `docs/tts-registry.json` 的 `requiredCount` 一致 | 含课文 + **00 入門** 五十音/寒暄等 |
| **缺失 MP3** | **必须为 0** | 每条 `data-tts-key` ↔ `tts-cache/{key}.mp3` |
| 根目录 `tts-cache/` | 必须存在且已提交 Git | 公网离线朗读靠此目录，不是 CDN 现拉 |
| `发布包/tts-cache/` | 必需条 0 缺失 | 由 `build-tts-cache.py` 写入双目录 |

**结论模板**（Agent 交付时必须写进回复）：

```text
语音对账：需朗读 N 条 · 缺失 0 · 报告 docs/TTS-对账报告-最新.md
```

若缺失 > 0：**禁止**声称「语音已齐」或发最终公网链接。

## 二、为什么你会觉得「每次都在丢、反复加载」？

**文件夹树状结构没变**，问题通常**不是** MP3 从磁盘消失，而是下面五类（按出现频率排序）：

### 1. 没用 HTTP 打开（最常见）

| 错误做法 | 结果 |
|----------|------|
| 双击 `index.html` → `file://` | `fetch(tts-cache/…)` 失败、录音失败、iframe「App 未加载」 |
| 8765 服务未启动就刷新预览 | 开发者竖屏里红字「iframe 内 App 未加载」 |

**正确**：`打开本地预览.bat` / `开发者竖屏验收.bat` → `http://127.0.0.1:8765/index.html?v=当前CACHE_VER`

### 2. 公网仓库里没有 MP3（push 漏文件）

- 本机 `tts-cache/` 有 1600+ 个文件，但 **commit 未包含** → GitHub Pages 上 404 → 提示「朗读失败」。
- **不是**「加载过一次又丢了」，而是**远程从未有过**。

**避免**：push 前 `git status` 看 `tts-cache/`；跑 `发布前自检.bat`（内含对账）。

### 3. 「反复加载」≠ 包丢了（机制说明）

语音是**完整嵌入包**：`tts-cache/{hash}.mp3`，与 JS 同仓，**不**按句向服务器买流量。

用户侧仍可能感觉「每次都要加载」，原因是：

| 机制 | 说明 |
|------|------|
| 每句一个文件 | 1100+ 条日文 → 1100+ 个 MP3；**第一次点某句** 浏览器要拉对应文件（HTTP 缓存后同设备再点通常不再下） |
| `?v=104` 缓存号 | 只让浏览器**换 JS/CSS**，**不会**删除已缓存的 MP3 |
| `warmPhrases` 预热 | 进入某关会后台 `audio.load()` 多条，像「又在下载」——是为减少点喇叭等待，不是丢包 |
| 微信内 | 在线 Google TTS 基本不可用，**必须**靠 `tts-cache/`；缺文件就只能失败 |

**产品承诺（文案可对用户说）**：  
第一次打开某课/某句可能等 1～2 秒；**同一手机、同一域名**下再点同一句，应走本地缓存，**不重复耗流量**（除非用户清缓存或换域名）。

### 4. 五十音「卡死、自己蹦」——不是显卡

- 原因：**大量 MP3 预热** + 滚动误触假名弹层（`IntroKanaTip`）。
- 代码侧：入門 `SpeakUI.bind(..., { skipPrefetch: true })`；滚动 600ms 内禁止弹层。
- 若仍白屏：先查 **8765 是否活着**，不是查缺 MP3。

### 5. 新增/改了日文，没重新生成包

- 改了 `lessons-mvp.js` / `intro-kana-tips.js` 等 → 新 `key` → **旧包没有对应 MP3**。
- **不是**随机丢失，是**需求清单变了**。

**修复**：`生成语音包.bat` → `批量检查语音包.bat`（缺失 0）→ `发布前自检.bat`。

## 三、发布前强制审核清单（必须逐项打勾）

**执行入口（二选一，等价）**：

```bat
发布前自检.bat
```

```bash
python scripts/pre-ship-check.py
```

| # | 检查项 | 通过标准 | 脚本/证据 |
|---|--------|----------|-----------|
| 1 | 语音编号对账 | `缺失 MP3 = 0` | `audit-tts-registry.py --write` |
| 2 | 对账报告归档 | `docs/TTS-对账报告-最新.md` 时间与本次一致 | `report-tts-audit.py` |
| 3 | 清单 JSON | `docs/tts-registry.json` 已更新 | 同上 `--write` |
| 4 | 本地 HTTP | `127.0.0.1:8765` 可打开 index | `pre-ship-check` → `check_local_http` |
| 5 | 抽样 MP3 可访问 | 8765 上 `/tts-cache/{key}.mp3` 返回 200 | `pre-ship-check` → `check_tts_mp3_http` |
| 6 | 版本号一致 | `index.html ?v=` = `CACHE_VER` | `pre-ship-check` |
| 7 | push 含 mp3 | `git diff --stat` 含新增 mp3 或确认无新句 | 人工 + status |
| 8 | 禁止 file 交付 | 交付说明写 localhost 链接 | [发布前自检.md](./发布前自检.md) |
| 9 | 入門专项 | 入門页不批量 prefetch；点字不滑表误触 | 代码约定 + 手测滑表 |
| 10 | 冒烟 | 第1课会話点 🔊 能出声 | 开发者竖屏验收 |

**任一项 FAIL → 不得对外发「最终版」链接。**

## 四、与目录结构的关系

```text
日语学习/
  tts-cache/           ← GitHub Pages / 本地 8765 朗读（主包，必须齐）
  发布包/tts-cache/    ← U 盘副本（gitignore，由生成脚本同步）
  docs/
    tts-registry.json          ← 机器清单 seq ↔ key ↔ line
    TTS-对账报告-最新.md       ← 每次发布前生成
  js/speech-engine.js          ← ttsCacheKey 算法真源
  scripts/tts_lib.py           ← 与 JS 对齐的扫描/对账
```

**树没变**，变的是 **JS 里的日文句子** 和 **是否把 mp3 提交到远程**。  
防复发靠 **脚本对账 + 发布前自检**，不靠人工点喇叭。

---

## 五、Agent / 开发者纪律（写进交付回复）

1. 改任何含日文朗读的 `js/data/*.js` 后 → 跑 `生成语音包.bat` → 对账缺失 0。  
2. 交付用户前 → 跑 `发布前自检.bat`，复制脚本末尾「交付反馈块」。  
3. 用户报「朗读失败」→ **先问是否 localhost:8765**，再查对账报告，不要一句句手试。  
4. 用户报「五十音卡」→ **先确认服务与 iframe**，再查入門 prefetch/误触，不要归因显卡。
