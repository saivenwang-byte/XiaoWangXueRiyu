# 教学类应用制作范式（我们 · 初稿）

> 从「标日课后巩固」项目讨论提炼，适用于 **学生 + 家长 + 学校** 共学场景的外语/学科类 H5 或小程序。

---

## 0. 动作前置：讨论优先（铁律）

| 规则 | 说明 |
|------|------|
| **先讨论** | 用户发来需求、截图、线框 → Agent **先** 复述理解、列待拍板项 |
| **后动手** | 用户明确「可以开工 / 同意 / 按这个做」后，才改代码与数据 |
| **禁止** | 看到文字就默认理解正确并直接实现 |
| **文档** | 详见 `过程讨论内容/00-讨论优先铁律.md` · Cursor 规则 `discuss-before-act.mdc` |

与 **文递自归**（对齐已冻结基线）并列，**两者都满足** 才可交付改动。

---

## 1. 产品伦理（先定调再写码）

| 原则 | 做法 |
|------|------|
| 学习优先 | 主路径 = 可验证的听说读测；趣味在 **通关之后** |
| 共同称谓 | 团队用「我们」；Agent = 产品经理 + 体验官，不替代教师判断 |
| 课本同构 | 目录、单元、课序与 **教材规划表** 一致，不用「跳课样本」当首页 |
| 冻结种子 | 先做好 1～3 课 **完整四关种子**，再复制课次，少改课内、多改目录 |
| 满分门槛 | 星级/成就与 **100%** 绑定；不把「点过」当学会 |

---

## 2. 信息架构三层

```text
层1  目录/地图     →  我在哪、还剩什么（四岛路径、单元、00入門）
层2  课内四关      →  我怎么学（単語·会話·文法·テスト）
层3  マイ/督导     →  我学得怎样（工作量、评估、追查、建议）
```

**禁止**：把 8 课平铺成「像全书已上线」；**推荐**：地图示意 + 分区板块 + 课序节点。

---

## 3. 数据与工程约定

| 块 | 约定 |
|----|------|
| `curriculum-catalog.js` | 课序、单元、released、地图站点 **唯一真源** |
| `lessons-mvp*.js` | 课内正文；冻结课 **不改** |
| `mvp-storage` | 进度、错题、日后 `dims`/`seaStars`/`story`/`dailyLog` |
| 发版 | `CACHE_VER` 递增；`pre-ship-check` 再 push |

---

## 4. 游戏化边界

| 用 | 不用 |
|----|------|
| 海星/海胆、路径节点、隐藏 00、漫画收藏 | 萝卜等未统一隐喻 |
| 课毕插画 → 单元连张 → 故事书 | 未三星即给奖励 |
| 写 **永不** 进星级 | 85 分及格线 |

---

## 5. 制作新教学 App 的检查清单

1. 是否有 **教材课序表** 真源？  
2. 首页是否是 **路径/地图** 而非课号超市？  
3. 种子课是否 **四关可冒烟** 再批量播种？  
4. 家长能否在 **マイ** 看到进度与薄弱点？  
5. 趣味资产（插画）是否 **机制先行、美术分期**？  
6. 是否与 `iteration-baseline` / 会议纪要 **可追溯**？
