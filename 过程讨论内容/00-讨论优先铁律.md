# 00 · 讨论优先铁律（动作前置条件）

**状态**：✅ 我们拍板 · 2026-05-21  
**级别**：**铁律** — 优先于任何「看到需求就写码」的惯性。

---

## 1. 规则正文

| 步骤 | 我们做什么 |
|------|------------|
| **① 接收** | 用户发来文字、截图、参考图、语音转写等 |
| **② 讨论** | Agent **先** 复述理解、对齐歧义、给出方案或线框解读；**不** 默认已全部理解 |
| **③ 确认** | 用户明确表示可以开工（如：「按这个做」「开工」「同意」） |
| **④ 动作** | 再改代码、数据、样式、发版 |

**禁止**：跳过 ②③，见到新描述就直接改仓库。

**允许不经确认即做的例外**（须在一句话内说明原因）：

- 用户 **明确点名** 只写文档 / 只改纪要 / 只加规则；
- 修复 **上一轮已确认方案** 内的明显笔误（且不改范围）。

---

## 2. 为什么

- 教学 App 的 **首页、阶段拆分、地图落点** 都强依赖产品意图，误读成本高。
- 我们约定 Agent = **产品经理 + 体验官**，不是「自动执行器」。

## 4. 与「文递自归」的关系

- **文递自归**：改代码前对齐 **已冻结基线**（课内、CACHE_VER、iteration-baseline）。
- **讨论优先**：改代码前对齐 **本轮用户意图**（尚未写入基线的新想法）。

两者 **都满足** 后才能动手。
